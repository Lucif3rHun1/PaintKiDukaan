WINNER — frontend print subsystem has two printReceipts

A function called printReceipt is exported from two files. src/pos/print.ts:354 saves a PDF to disk via doc.save(). src/pos/sales/printReceipt.ts:120 exports printSaleReceipt which is aliased to printReceipt on import.

Deepening — three coherent changes, not six

1. Split src/pos/print.ts by output type

The current file is the classic "kitchen sink" — three independent output types (label, receipt, return) plus a money util plus stock-size constants. Each gets its own file; nothing else changes.

src/pos/printing/
├── labels.ts          printLabel, LabelSpec, BatchLabel, PrintConfig, ThermalSize, THERMAL_SIZES, LOCKED_FORMAT
├── receipts.ts        ReceiptSpec, buildReceiptPdf, buildReceiptPdfBlob, printReceipt
├── returns.ts         ReturnReceiptSpec, buildReturnReceiptPdf, buildReturnReceiptPdfBlob
└── money.ts           paiseToRupees, formatRupeesFromPaise

src/lib/print.ts deletes. src/pos/print.ts becomes a barrel that re-exports the four files — same external surface, internal split is the seam. Files that import {printLabel} from ../../pos/print keep working unchanged.

2. Extract the routing decision into PrintRouter

The "thermal vs PDF" decision is currently three lines of if buried in printSaleReceipt, copy-pasted into printReturnReceipt. Promote it to a typed function the two callers share, and accept a PrintEnvironment so the platform check is no longer a hidden global.

// ponytail: this is the only seam — the routing decision is currently hidden inside
// printSaleReceipt and copy-pasted into printReturnReceipt. Extract once.
export type PrintEnvironment = {
  readonly os: "windows" | "macos" | "linux";
  readonly configuredPrinter: string | null;
  readonly paperSize: "thermal-58mm" | "thermal-80mm" | "a4" | "a5" | null;
};

export type PrintRoute =
  | { kind: "thermal"; printer: string }
  | { kind: "pdf"; pdf: Blob };

export function decideRoute(
  env: PrintEnvironment,
  payload: EscPosPayload,
): Promise<PrintRoute>;   // throws if neither is viable

// printSaleReceipt shrinks to:
const route = await decideRoute(env, payload);
return route.kind === "thermal"
  ? { destination: "thermal" }
  : await printReceiptPdfBlob(route.pdf);   // identical for sales & returns

The dev-PDF fallback stays in printOrDownload.ts — it's an orchestration concern, not a routing one. printSaleReceipt becomes 12 lines.

3. Split printOrDownload.ts by intent

Five verbs in one file is the smell. Each verb becomes a one-file module that takes the receipt route as input.

src/pos/sales/printing/
├── print.ts           printSaleById(saleId) → toasts "sent to {printer}" or "PDF saved: {path}"
├── download.ts        downloadSalePdfById(saleId) → triggers browser download
├── share.ts           shareSalePdfById(saleId) → navigator.share or download fallback
└── returns.ts         safePrintReturnById(returnId) → wraps printReturnReceipt

The settings-load helper (loadSettings()) and sale-lookup helper move into a single src/pos/sales/printing/_load.ts — four functions share it, but the file is 40 lines and clearly named. SalesPage imports one verb per call site:

// Before (in SalesPage.tsx):
await safePrintSaleById(saleId);

// After — same surface, but the "safe" wrapper can be dropped because
// each verb now owns its own try/catch+toast:
await print.print(saleId);          // toasts internally
await print.download(saleId);        // toasts internally
await print.share(saleId);           // toasts internally

End state: src/pos/print.ts is a barrel of four files, not a 787-line god file. printSaleReceipt is 12 lines, not 54, and the fallback chain is decideRoute()'s problem. printOrDownload.ts is gone — its five verbs each have a home. The 15 files that import from pos/print change zero import paths.

Audit passes — candidates considered, not picked

Three of the seven scanned modules had real smells. Each one had a reason not to deepen this pass — the printed receipt is the one a cashier actually watches fail.

src-tauri/src/db/inline_migrations.rs medium

25 sequential m_inline_NNN(conn)?; calls in run_inline_migrations

The shape (sequential ?-chained function calls, 25 of them) mirrors what the auto-updater looked like in v0.3.3. But each migration is 5–15 lines, idempotent by construction (every one gates on a has_col or table_exists check before mutating), and has exactly one caller (Db::open at boot). Replacing m_inline_001(conn)?; with for m in MIGRATIONS { m(conn)?; } doesn't add a test seam — there's no test you'd write that catches "migrations ran in wrong order". Ponytail: this exists, skip the abstraction. Add a real migration runner when a 26th migration forces it.

src-tauri/src/db/mod.rs::open low

7 mixed steps: key → cipher → schema-bootstrap → migrations → ANALYZE → PRAGMAs

Real smell — a single 80-line function that does key handling, cipher configuration, schema bootstrap (fresh vs wipe-and-rebootstrap), inline migrations (the first two, the rest are in the sibling file), and final ANALYZE. But there is exactly one caller (lib.rs boot), and the order of the seven steps is load-bearing — extracting a DatabaseBootstrap trait just pushes the same sequence into a wrapper struct. Save for the next pass, or fold into the inline_migrations split if it ever happens.

src/pos/sales/SalesPage.tsx low

1,294 lines, 45 symbols — the workhorse POS page

Definitely a candidate for the next pass: state is jammed into the page component (kind, customer, lines, splits, validity, ackFlag, recent, busy, saleUnits, lastUsedUnit, three dialog opens, customerTypes, editMode, draft, draftStatus), and the useEffect for edit-mode restore is 40 lines. But the page does what AGENTS.md documents, has a passing test (SalesListPage.test.tsx), and is the workhorse — refactoring it would block every concurrent feature branch. Splitting into <SalesForm> + <RecentSales> + <SplitPaymentPanel> is the right move but it's a separate pass.

Grilling prompt — to make this design final

Open these with the user before implementing:

  1. Are we keeping src/pos/print.ts as the public path, or renaming to src/pos/printing/? Renaming forces 15+ files to update paths. Keeping pos/print.ts as a 4-line barrel means zero imports move — the split is internal. The barrel is uglier but zero-risk for the next release.
  2. Where does PrintEnvironment get built? Currently isWindows() reads navigator.platform directly. Should the env come from useSecurity() (already initialised), a new usePlatform() hook, or stay inline at the call site?
  3. Does the dev-PDF fallback (macOS cmd_print_receipt_dev) ship to production? It's the only path a macOS cashier sees today — Windows + thermal printer is the only path that ever hits printEscPosReceipt. If macOS is a real deployment target, the dev fallback is load-bearing, not dev-only. The router should make that explicit.
  4. Share intent — Safari iOS support? navigator.share is iOS Safari's bridge to AirDrop / Files / Mail. If a paint-store cashier on an iPad shares a receipt via AirDrop to the customer's phone, the PDF blob path matters more than the toast.
  5. What does usePrinter().discover() return today? discoverSystemPrinters is called from HardwareSettings, but printSaleReceipt only reads getDefaultPrinter("receipt"). Are we sure the configured default is the right one at print time, or should printSaleById re-discover?
