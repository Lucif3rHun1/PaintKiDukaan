🔴 HIGH PRIORITY: Extract Shared Types Module

Module: lib/types.ts (new) — consolidate Sale, PaymentMode, CustomerLedger, VendorOutstanding, Role, User, Session, Bootstrap

Why: 8+ type definitions are duplicated across 3 files. The Sale type lives in pos/types.ts but domain/types.ts and domain/ipc.ts import it, creating a B→C cycle that violates the slice dependency graph. Every touch to Sale fields requires coordinated edits across both slices.

Impact: Breaks 6 domain→pos import violations, eliminates 8 duplicate type definitions, and makes the slice boundaries enforceable by module resolution alone.

--- ### Shallow Modules (Problematic — Deepen or Remove) A **shallow module** has an interface almost as wide as its implementation. Pass-through re-exports, thin wrappers, and barrel files that add indirection without adding behavior. The caller learns the module's API but gains almost nothing in return. ---
🧮 domain/formulas/api.ts 🟡 SHALLOW — Deepen or Remove
InterfaceDepthProblem
9 functions total:
6 re-exported from domain/ipc.ts
3 defined locally		 Near zero for the re-exports. Thin wrappers for the local 3. Deletion test: callers could import from domain/ipc.ts directly. The real API lives there. This module adds a layer of indirection with no added behavior.

Fix: Either move the 6 domain/ipc.ts functions here (making formulas/api.ts the canonical entry point) or delete it and re-point imports.

---
📦 domain/index.ts 🟡 SHALLOW — Deepen or Remove
InterfaceDepthProblem
Barrel file mixing:
types + components + namespaces		 Zero. Pure re-exports. Conflates type definitions with component exports and namespace declarations. Tree-shaking-unfriendly. Forces consumers to depend on the entire domain module.

Fix: Delete the barrel. Let consumers import from specific files (domain/items/api, domain/types).

InterfaceDepthProblem

Fix: Standardize on the domain/ipc.ts pattern (AppError normalization + correlation IDs) across all slices.

Fix: Adopt the domain/ipc.ts pattern consistently, or consolidate all IPC wrappers into a single typed invoke module.

The differing `Session` and `Bootstrap` types are silent bugs waiting to happen, callers may assume compatibility where none exists. --- ## Three IPC Patterns The codebase has three inconsistent approaches to calling Tauri commands from TypeScript: ### Pattern 1: `domain/ipc.ts` — AppError Normalization + Correlation IDs ✅ ```typescript // domain/ipc.ts — THE STANDARD import { tauriInvoke } from '@/lib/security/tauri'; import type { AppError } from '@tauri-apps/api/core'; export async function invokeWithTypedError( cmd: string, args?: Record ): Promise { // Wraps tauriInvoke, normalizes errors to AppError, // attaches correlation ID for debugging } ``` ### Pattern 2: `pos/api.ts` — isTauri() Browser Fallbacks ⚠️ ```typescript // pos/api.ts — NO ERROR NORMALIZATION import { isTauri } from '@tauri-apps/api/core'; export async function createSale(args: CreateSaleArgs): Promise { if (isTauri()) { return invoke('cmd_create_sale', { args }); } // Browser fallback — mock data or throw } ``` ### Pattern 3: `shell/lib/ipc.ts` — Raw Invoke ⚠️ ```typescript // shell/lib/ipc.ts — NO ERROR NORMALIZATION, NO CORRELATION import { invoke } from '@tauri-apps/api/core'; export const ipc = { getSettings: () => invoke('cmd_get_settings'), saveSettings: (s: Settings) => invoke('cmd_save_settings', { settings: s }), // ...30+ raw wrappers }; ``` **Fix:** Standardize on Pattern 1 (domain/ipc.ts). Its AppError normalization means callers get consistent error shapes. The correlation ID makes production debugging tractable. Pattern 2's `isTauri()` fallback is useful but should wrap Pattern 1, not replace it. Pattern 3 is the highest-risk: raw invoke with no error typing means any backend error surfaces as a string, losing context. --- ## Testability Friction | Module | Friction | Root Cause | Improvement Path | |--------|:--------:|------------|------------------| | `auth.rs` | 🔴 HIGH | Global `AppState` + crypto init + session management all coupled | Split into state, crypto, session, commands modules | | `db::open()` | 🔴 HIGH | SQLCipher initialization, key derivation, migration runner all in one function | Extract key derivation; test migrations separately with temp DBs | | `App.tsx` | 🟡 MEDIUM | God component: 12+ routes, 8 modals, activity tracking | Decompose into independently testable sub-components | | `session.rs` | 🟡 MEDIUM | Global `static Mutex` for session state | Inject session state as a parameter, test with in-memory state | | `security/mod.rs` | 🟡 MEDIUM | 33 Windows-only submodules compiled on all platforms | Use `cfg` attributes more granularly; extract platform-independent logic | | `sales.rs` | 🟡 MEDIUM | Business logic mixed with Tauri command handlers | Extract pure business logic into testable functions, keep commands as thin wrappers | | `day_close.rs` | 🟡 MEDIUM | Report generation coupled to DB queries | Separate query layer from report formatting | | `pos/api.ts` | 🟢 LOW | Wrapper functions are trivially testable, but `isTauri()` fallbacks add branching | Mock `isTauri()`, test both branches | **Key observation:** The Rust backend's testability friction is concentrated in modules that own global state.
